## 用户需求

通过查阅本周（2026-06-08 ~ 2026-06-12）群组复盘记录，从代码层风控硬约束和Prompt/SOP流程两个维度进行系统全面改进。

## 复盘暴露的核心问题

1. **「浮盈→亏损」模式跨两周复现**：东睦股份+4.74%浮盈最终-2.44%止损（-198元）、厦门钨业+2.9%浮盈最终-0.31%止损（-21元）、上周振华科技-184元、巨化股份-246元。新SOP的铁律二（两道移动止盈保护）已设计但完全依赖AI自觉执行，无代码层强制保护。
2. **仓位决策层系统性脱节**：Pi反复给出40-60%仓位上限，但交易系统实际仅使用建议上限的13-17%。大行情来临时系统的仓位响应机制不确定。
3. **选股方向偏离最强主线**：主线捕捉率仅33%——周二半导体+96亿却买入机械基件东睦，周五有色金属+144亿却空仓度过。
4. **V反/假突破辨别能力不足**：周五10:50和13:31两次误判V反为趋势修复，是本周两笔亏损的技术根源。
5. **止损依赖扫描窗口存在结构性滞后**：东睦4分钟内滑出0.74个百分点超出止损线，系统完全依赖每小时定时窗口检测止损。
6. **缺乏"最多拒绝N次"的终止边界**：周二8次拒绝假突破无上限定义，如果第9次也是假突破怎么办？

## 代码架构层面发现的缺口

1. **风控规则代码层与Prompt层严重不一致**：`check_risk()`仅做3项基础检查（资金充足性、单笔最大40%、卖出持仓存在性），Prompt中详尽的铁律二移动止盈、动态止损、回撤熔断、T+1规则在代码层全部缺失。
2. **回撤熔断无代码层实现**：Prompt要求"总回撤≥5%停止所有买入"，但`check_risk()`不查询账户总盈亏，代码中没有任何位置执行此规则。
3. **T+1约束无代码层硬保护**：完全依赖AI自觉遵守，如果Pi在closing窗口错误卖出今日买入的股票，代码层不会拦截。
4. **无实时持仓止损监控**：系统完全依赖每小时定时窗口检测止损条件，股价在窗口之间暴跌时止损最多延迟1小时。

## 技术栈

- **Backend**: Python 3.x + FastAPI + APScheduler，交易层使用 SQLite (trades.db) + PaperTradingEngine
- **Pi Server**: Node.js/TypeScript，通过 HTTP 调用 Backend API，使用 DeepSeek LLM + Function Calling 自主决策
- **策略链**: StrategyChain 类管理盘前→盘中→交易→反馈的完整状态流转，通过 JSON 文件持久化
- **配置**: YAML (+ config.yaml) + .env

## 实现方案

### 总体策略

采用**双层防护**架构：代码层提供硬约束（拒绝/熔断/强制执行），Prompt层提供软约束（决策逻辑/SOP流程），代码层拦截优先于Prompt层决策。当代码层检测到违规时直接拒绝交易并反馈给Pi，Pi可根据拦截原因调整决策后重试。

### 关键设计决策

1. **止损监控线程使用APScheduler已有的BackgroundScheduler机制**，在`lifespan`中启动/停止，轮询间隔30秒，复用已有的`MarcusVNPyExecutor.get_account()`和雪球行情价格查询路径
2. **回撤熔断在`check_risk()`入口处执行**，每次buy()调用前检查账户总回撤，达到5%直接拒绝并写入风控日志
3. **T+1拦截在`check_risk()`的sell分支中执行**，查询trades.db中该symbol今日是否有买入记录，有则拒绝
4. **仓位利用率检查在`_execute_pi_trade()`解析SIGNAL后、发送给Pi Server前执行**，计算Pi建议仓位vs实际已用仓位的偏离率，偏离>60%时在Prompt中注入警告
5. **拒绝次数上限在Prompt中增加规则**，Pi在单窗口连续拒绝N次后终止当日该窗口交易，同时代码层记录拒绝计数

### 性能考量

- 止损监控线程30秒轮询一次：持仓≤5只时每次轮询约2-3秒（串行调用雪球行情API），对系统CPU和内存影响可忽略
- `check_risk()`新增的T+1查询和回撤计算均为SQLite本地查询，延迟<10ms
- 仓位利用率检查仅读取StrategyChain的JSON文件，无数据库IO

### 向后兼容

- 所有`check_risk()`新增检查均返回明确拒绝原因，不影响原有`allowed=True`的路径
- 止损监控线程可被配置开关控制，默认启用但在配置文件中可禁用
- Prompt修改保持与现有SOP结构一致，仅增加新规则段落，不删除已有内容
