## Context

当前 `StopLossMonitor` 的成本止损（规则 0b）使用固定阈值：从未盈利 -4%、小盈转亏 -3%、无 HWM -6%。板块背离止损（规则 1）使用固定阈值 -3pp。止损执行层（`_execute_stop`）有早盘冷静期和 T+1 保护，但没有 60 分钟趋势校验。

豫能控股 07-20 卖飞事件同时暴露了三个缺陷：(1) 12.37% 振幅的股票用了固定止损线，正常波动就触发；(2) 60 分钟趋势未破（MA10 > MA30）却被卖出；(3) 极端恐慌日全市场流出 -372 亿，电力板块资金在净流入，规则 1 的 -3pp 阈值产生假阳性。

三个缺陷需要同时修复——单独修任何一个，另外两个漏洞还在。

## Goals / Non-Goals

**Goals:**
- P0a: 规则 0b 三档（从未盈利/小盈转亏/无HWM）全部动态化，基础阈值不同、系数不同
- P0b: 止损执行前校验 60 分钟趋势，MA10 > MA30 时暂停执行
- P0c: 规则 1 在极端恐慌日（全市场流出 > 300 亿）自动放宽阈值
- 改动保持在各自方法内部，不改变规则优先级和主循环

**Non-Goals:**
- 不改动规则 0a（破底）、2（铁律二）、2.5（技术背离）、2.6（60分钟分析）、3（大盘动态）
- 不创建独立的 L0 元规则层（等有 ≥2 条规则需要时再提取）
- 不引入仓位档位感知（留给后续 P1 change）
- 不修改 API 接口或数据库 schema

## Decisions

### Decision 1: P0a — 三档均动态化，系数差异化

**选择**：所有三档都使用 `max(基础阈值, 振幅×系数)` 公式，但系数不同。

| 场景 | 基础阈值 | 系数 | 振幅 3%（低波） | 振幅 12%（高波） |
|:---|:---:|:---:|:---:|:---:|
| 从未盈利 | -4% | 0.40 | max(-4%, -1.2%) = **-4%** | max(-4%, -4.8%) = **-4.8%** |
| 小盈转亏 | -3% | 0.40 | max(-3%, -1.2%) = **-3%** | max(-3%, -4.8%) = **-4.8%** |
| 无 HWM | -6% | 0.50 | max(-6%, -1.5%) = **-6%** | max(-6%, -6.0%) = **-6.0%** |

**关键设计**：小盈转亏和从未盈利使用相同系数（0.40），仅在基础阈值上差 1pp。这解决了之前 change 的悖论——Stock A（曾盈 3.1%）不会因为"保本意愿更强"反而得到更紧的止损。两档在低波时保持现有差异（-3% vs -4%），高波时同比例扩宽。

**备选方案**：只扩宽"从未盈利"——会产生悖论（曾盈 3.1% 的股票比曾盈 2.9% 的股票更早被止损）。已否决。

### Decision 2: P0b — 60 分钟趋势校验放在执行层

同之前 design，无变化：`_execute_stop` 开头增加 MA10/MA30 判断，3 秒超时，同日同股最多暂停 3 次。

### Decision 3: P0c — 架构 A（规则内嵌环境感知），不提取 L0 层

**选择**：在 `_check_sector_divergence` 内部增加环境感知逻辑，不创建独立的元规则层。

**理由**：目前只有规则 1 需要极端行情阈值调整。规则 0b 靠振幅动态化解决，规则 3 已有市场感知。等 P1（试探仓趋势止损）上线后，如果趋势信号也需要市场状态感知，再提取 L0 层。当只有一条规则需要时，架构 A 是正确的抽象层次。

**备选方案**：架构 B（独立 L0 层）——需要定义跨规则的 params 接口，目前过度设计。已否决。

### Decision 4: P0c — 两步决策树 + 个股防线

```
触发门: 全市场主力净流出 > 300 亿

STEP 1: 个股自身主力资金（防真出货）
    ├── 个股主力净流出 > 0.3 亿（或净占比 < -10%）
    │     → 判定为真出货，阈值保持 -3pp，不调整
    └── 否则 → 进入 STEP 2

STEP 2: 板块资金强度分档
    ├── 板块主力净流入 > 10 亿  → 阈值: -3pp → -8pp（高置信度错杀）
    ├── 板块主力净流入 0~10 亿  → 阈值: -3pp → -5pp（中等置信度错杀）
    └── 板块主力净流出          → 阈值: 保持 -3pp（无错杀基础）
```

**数据源**：
- 全市场资金：`get_moneyflow_mkt()` → net_amount（万元），已在 `/api/v1/market/moneyflow-mkt` 可用
- 个股资金：`get_stock_moneyflow(symbol)` → main_net（元），已在 `/api/v1/market/moneyflow/{symbol}` 可用
- 板块资金：`get_concept_fund_flow()` → main_net（万元/板块），已在 `/api/v1/market/concept-fund-flow` 可用

**备选方案**：不检查个股主力资金（STEP 1）——可能把真出货误判为错杀。已否决。

### Decision 5: 规则 2（铁律二）在极端恐慌日不收紧

**选择**：铁律二在极端恐慌日保持原样，不收紧、不放宽。

**理由**：恐慌尖底的价格是流动性挤压的扭曲价格。在扭曲价格上收紧铁律二 = 在最低点锁利 = 把浮盈变成实际亏损。这与 P0a/P0c 的方向一致——极端行情下止损/止盈都应收敛而非激进。

## Risks / Trade-offs

- **[Risk] 个股资金流调用可能增加止损检查延迟** → `_check_sector_divergence` 本身已有 HTTP 调用，增加一次个股资金流查询。设置 2 秒超时，超时则跳过 STEP 1（宁可漏过防线也不阻塞止损）
- **[Risk] 全市场流出 300 亿阈值可能需要调优** → 先硬编码在代码中，后续从回测数据中校准。初期保守一些（阈值设高一点，减少误触发）
- **[Risk] P0a 小盈转亏也扩宽可能导致 -3% 变太宽** → 0.40 系数在振幅 12% 时才到 -4.8%，振幅 8% 时还是 -3.2%，基本保持现有纪律。只有真正的高波股才显著扩宽
- **[Trade-off] 不提取 L0 元规则层** → 如果后续规则 0b 或规则 2 也需要极端行情感知，会产生重复代码。但目前只有规则 1 需要，重复代码的量（一个 if 块）远小于过早抽象的代价
